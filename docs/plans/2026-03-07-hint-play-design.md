# 出牌提示功能设计

**日期：** 2026-03-07

**目标：** 在轮到当前玩家出牌时，增加一个“提示”按钮。点击后自动选中一组合法可出的牌，并且在同一局面下连续点击时，能够在全部可出方案之间稳定轮换。

## 背景

当前客户端只有手动点牌、`出牌`、`不出` 三个交互：

- `packages/client/src/components/Game/ActionBar.tsx` 负责出牌与不出按钮
- `packages/client/src/store/useGameStore.ts` 负责当前手牌与选牌状态
- `packages/shared/src/rules/validator.ts` 只能校验“某组已选中的牌是否合法”

项目目前没有“从手牌中枚举全部可出组合”的能力，因此无法稳定实现提示轮换。

## 需求边界

已确认的交互约束：

- 点击 `提示` 后，应自动选中一组可出的组合
- 在同一局面下连续点击 `提示`，需要在所有可出方案之间轮换
- 当普通解法存在时，轮换列表中仍然要包含炸弹和王炸
- 当前功能只影响前端选牌体验，不改动服务端协议

## 架构设计

### 1. shared 新增纯函数负责枚举提示

在 `packages/shared/src/rules/` 中新增一个纯函数，输入：

- 当前手牌 `hand`
- 当前桌面上一手牌 `previousPlay`

输出：

- 已按稳定顺序排序的 `CardPlay[]`

该函数只负责：

- 从当前手牌中枚举所有合法组合
- 基于现有牌型识别和比较逻辑过滤可出方案
- 对结果去重
- 生成稳定排序

该函数不负责：

- UI 轮换索引
- 当前是否轮到自己
- 选中状态管理

### 2. client 只负责轮换与选中

客户端不再自己推导规则，只消费 shared 返回的提示列表：

- `ActionBar` 增加 `提示` 按钮
- `useGameStore` 增加用于提示轮换的状态与 action
- 点击 `提示` 时，基于当前局面生成提示列表并选中下一项
- `PlayerHand` 继续通过 `selectedCards` 渲染高亮

### 3. 不改服务端

服务端已有出牌合法性校验，不需要知道客户端是否用了提示。提示只是一种前端快捷选牌方式，因此：

- 不新增 socket 事件
- 不新增服务端状态
- 不修改现有 `game:playCards` 流程

## 数据流

### 点击提示

1. 客户端判断当前是否为自己回合，且处于 `GamePhase.Playing`
2. 从 store 读取 `myHand`、`lastPlay`、`currentTurn`
3. 调用 shared 的提示枚举函数，得到排序后的 `CardPlay[]`
4. 使用 store 中的轮换索引选中下一组提示牌
5. 把该组 `CardPlay.cards` 写入 `selectedCards`
6. 下次点击同一局面下的 `提示` 时，索引递增并循环

### 局面变化时重置轮换

以下情况都应把提示轮换回到初始状态：

- 自己手牌变化
- `lastPlay` 变化
- 连续两家 `pass` 使 `lastPlay` 变为 `null`
- 当前轮次发生变化
- 新一局开始
- 收到 `game:syncState`

## 排序规则

为了让用户感知稳定，提示列表必须使用固定排序。

### 自由出牌

按如下顺序：

1. 普通牌型
2. 炸弹
3. 王炸

普通牌型内部按以下规则稳定排序：

- 先按牌型优先级
- 同牌型按 `mainRank` 从小到大
- 长度型牌（顺子、连对、飞机）先按 `length`，再按 `mainRank`
- 带牌型再按附带牌的点数字典序稳定排序

### 跟牌

按如下顺序：

1. 所有能压过上家的普通解法
2. 炸弹
3. 王炸

这样既满足“优先给普通解法”的直觉，也满足“炸弹/王炸必须纳入轮换”的需求。

## 去重策略

因为同一手牌在枚举过程中可能通过不同组合路径得到相同结果，所以需要去重。

去重键采用：

- `type`
- `mainRank`
- `length`
- 排序后的牌面序列

只保留排序后首次出现的结果，确保提示轮换稳定。

## UI 交互

### 按钮状态

- `提示` 按钮仅在 `phase === Playing && isMyTurn` 时可用
- 非自己回合时禁用
- 与 `出牌`、`不出` 放在同一操作区

### 无可出方案

理论上只会出现在“可以不出，但手里没有能压过上家的普通牌且也没有炸弹/王炸”的场景。

此时：

- 不修改当前选中的牌
- 给出错误提示：`没有可出的牌，请选择不出`

### 与手动选牌的关系

用户手动点牌后，不会破坏当前局面的提示能力。再次点击 `提示` 时：

- 直接按当前轮换结果覆盖 `selectedCards`
- 不要求用户先手动清空选牌

## 测试策略

### shared

为新增纯函数补充单测，覆盖：

- 自由出牌时能枚举常见组合
- 跟牌时只返回可压过上家的方案
- 普通解法与炸弹/王炸同时存在时，顺序正确
- 重复组合被去重
- 排序稳定

### client

补充前端测试，覆盖：

- 连续点击 `提示` 能轮换选中不同方案
- 局面变化后轮换重置
- 无解时设置错误提示

## 风险与约束

- 斗地主组合空间不小，但当前手牌上限只有 20 张，采用“按牌型模板枚举 + 现有规则校验”的方式即可控制复杂度
- 提示逻辑必须复用 shared 规则层，避免客户端单独维护另一套牌型判断
- 由于提示只影响选牌，不应改变现有出牌成功后的状态流转
